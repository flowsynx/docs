# FlowSynx Plugins API

The FlowSynx Plugins API allows clients to manage plugin lifecycle operations such as 
listing, installing, retrieving details, updating, and removing plugins. 
It uses RESTful principles and supports both **Basic Authentication** and **Bearer Token Authentication**.

## Authentication & Authorization
#### Supported Modes:
- **No Authentication** (optional / configurable)
- **Basic Authentication**
- **Bearer Token Authentication**
- **Authorization:**
  - Required **unless No Authentication is enabled**
  - Roles: `admin` or `plugins`

:::warning
Security note: **No Authentication** should only be enabled for development, internal networks, or behind a trusted gateway.
:::

### 1. No Authentication
When **No Authentication** is enabled, requests are accepted without credentials.

```bash
curl http://localhost:6262/plugins
```

- **Authentication:** ❌ Not required
- **Authorization:**  ❌ Not required
- **Recommended for:**
  - Local development
  - Internal services
  - Reverse-proxy–protected deployments

### 2. Basic Auth

```bash
curl -u username:password http://localhost:6262/plugins
```

- **Authentication:** ✅ Required
- **Authorization:**  ✅ Required (`admin` or `plugins` role)

### 3. Bearer Token
```bash
curl -H "Authorization: Bearer <your_token>" http://localhost:6262/plugins
```

- **Authentication:** ✅ Required
- **Authorization:**  ✅ Required (`admin` or `plugins` role)

## API Endpoints
### List Installed Plugins
- **Method**: `GET`
- **Endpoint**: `/plugins`
- **Body Required**: No,
- **Query Parameters (optional)**:
  - **page** (integer): Page number to retrieve. Default is 1.
  - **pageSize** (integer): Number of items per page. Default is 25.

#### `curl` Example
```bash
curl -X GET "http://localhost:6262/plugins?page=1&pageSize=25" \
  -H "Authorization: Bearer <your_token>"
```

#### Example response
```json
{
  "data": [
    {
      "id": "6c60c794-99b9-47e8-aa34-18a14cdc4583",
      "type": "FlowSynx.Execution.ExternalProcess",
      "version": "1.0.0",
      "description": "Execute external processes, scripts, or commands and capture their output, enabling integration of external tasks within workflows."
    }
  ],
  "messages": [],
  "succeeded": true,
  "pagination": {
    "currentPage": 1,
    "totalPages": 1,
    "totalCount": 1,
    "pageSize": 25,
    "hasPreviousPage": false,
    "hasNextPage": false
  },
  "generatedAtUtc": "2025-08-02T13:10:05.2517196Z"
}
```

:::note
- The pagination object provides metadata for paginated results:
  - **currentPage**: Current page number.
  - **totalPages**: Total number of pages available.
  - **totalCount**: Total number of plugins.
  - **pageSize**: Number of items per page.
  - **hasPreviousPage**: true if there’s a previous page.
  - **hasNextPage**: true if there’s a next page.

You can use the **page** and **pageSize** query parameters to retrieve specific pages of plugin results.
:::

### Install Plugin
- **Method**: `POST`
- **Endpoint**: `/plugins`
- **Body Required**: Yes
- **Content-Type**: `application/json`

#### Request Body
```json
{
  "Type": "FlowSynx.Execution.ExternalProcess:1.0.0"
}
```

The `Type` field uses a structured, versioned string format to uniquely identify an execution definition. 
It follows the pattern `<Namespace>.<Name>:<Version>`, where the namespace groups related execution types, 
the name identifies the specific execution implementation, and the version specifies the exact revision 
of that implementation. In the example `FlowSynx.Execution.ExternalProcess:1.0.0`, `FlowSynx.Execution` 
represents the logical grouping, `ExternalProcess` is the execution type identifier, and `1.0.0` is the 
semantic version, allowing precise resolution, compatibility control, and safe evolution of execution 
definitions over time.

#### `curl` Example
```bash
curl -X POST http://localhost:6262/plugins \
  -H "Authorization: Bearer <your_token>" \
  -H "Content-Type: application/json" \
  -d '{
        "Type": "FlowSynx.Execution.ExternalProcess:1.0.0"
      }'
```

#### Example response
```json
{
  "data": {},
  "messages": [
    "The plugin has been installed successfully."
  ],
  "succeeded": true,
  "generatedAtUtc": "2025-08-02T13:09:52.9549804Z"
}
```

### Update Plugin
- **Method**: `PUT`
- **Endpoint**: `/plugins`
- **Body Required**: `Yes`
- **Content-Type**: `application/json`

#### Request Body
```json
{
  "Type": "FlowSynx.Execution.ExternalProcess:1.0.0->1.1.0"
}
```

The `Type` field uses a version-mapping string format to describe both the **source** and **target** versions 
of an execution definition. It follows the pattern `<Namespace>.<Name>:<FromVersion>-><ToVersion>`, where the 
namespace groups related execution types, the name identifies the execution implementation, and the version 
range specifies an update path. In the example `FlowSynx.Execution.ExternalProcess:1.0.0->1.1.0`, `FlowSynx.Execution.ExternalProcess` 
identifies the execution type, `1.0.0` represents the currently referenced version, and `1.1.0` indicates the 
target version to which it should be updated, enabling explicit, controlled version transitions while preserving 
clarity and traceability in execution definitions.


#### `curl` Example
```bash
curl -X PUT http:/localhost:6262/plugins \
  -H "Authorization: Bearer <your_token>" \
  -H "Content-Type: application/json" \
  -d '{
        "Type": "FlowSynx.Execution.ExternalProcess:1.0.0->1.1.0"
      }'
```

#### Example response
```json
{
  "data": {},
  "messages": [
    "The plugin has been uninstalled successfully."
  ],
  "succeeded": true,
  "generatedAtUtc": "2025-08-02T13:09:52.9549804Z"
}
```

### Uninstall Plugin
- **Method**: `DELETE`
- **Endpoint**: `/plugins`
- **Body Required**: `Yes`

#### Request Body
```json
{
  "Type": "FlowSynx.Execution.ExternalProcess:1.0.0"
}
```

#### `curl` Example
```bash
curl -X DELETE http://localhost:6262/plugins \
  -H "Authorization: Bearer <your_token>" \
  -H "Content-Type: application/json" \
  -d '{
        "Type": "FlowSynx.Execution.ExternalProcess:1.0.0"
      }'
```

#### Example response
```json
{
  "data": {},
  "messages": [
    "The plugin has been uninstalled successfully."
  ],
  "succeeded": true,
  "generatedAtUtc": "2025-08-02T13:09:52.9549804Z"
}
```
